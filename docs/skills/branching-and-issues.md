# Branching and issues

## Hard rules
- Every implementation branch maps to one issue.
- Branch name format: `type/issue-area-slug`
- No direct commits to `main` or `release/*`
- No silent scope expansion inside a PR
- If work grows, open a follow-up issue

## Allowed types
feat, fix, docs, ci, chore, hotfix, refactor, test

## PR rules
- PR must link an issue
- PR must state in-scope / out-of-scope
- PR must list test evidence
